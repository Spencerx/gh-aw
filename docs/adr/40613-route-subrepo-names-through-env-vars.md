# ADR-40613: Route Sub-Repo Names Through Env Vars in Generated Git Credentials Step

**Date**: 2026-06-21
**Status**: Draft

## Context

The compiler's `GenerateConfigureGitCredentialsSteps` (in `pkg/workflow/checkout_step_generator.go`) generates a "Configure Git credentials" step for multi-repo (`target-repo`) workflows. Prior to this change, each sub-repo's repository name was inlined directly into the `git remote set-url` shell command. When a workflow used a `workflow_dispatch` input expression such as `${{ github.event.inputs.target_repo }}` as the repository name, the compiler emitted that expression unescaped inside the `run:` block. The template-injection scanner correctly flagged this pattern as unsafe — but here the offending expression was the compiler's *own* generated code, which blocked all multi-repo `target-repo` workflows from compiling on v0.80.6. A way was needed to expand the expression at runtime without ever placing `${{ }}` inside a shell command string.

## Decision

We will assign each sub-repo checkout a dedicated environment variable (`GH_AW_SUBREPO_0`, `GH_AW_SUBREPO_1`, …) in the generated step's `env:` block, and reference that variable (`${GH_AW_SUBREPO_N}`) from the `git remote set-url` command instead of inlining the repository name. Expression-based repository names are written unquoted into the `env:` block so the GitHub Actions runner expands them; literal names are double-quoted as safe YAML scalars. As a result, no `${{ }}` expression ever appears inside a `run:` block, so the template-injection scanner no longer rejects the compiler's own output while runtime substitution still works.

## Alternatives Considered

### Alternative 1: Exempt compiler-generated expressions from the scanner
Add an allow-list or suppression so the template-injection scanner ignores expressions known to originate from the compiler. Rejected because it weakens a security control globally, is fragile to maintain, and still leaves a real `${{ }}`-in-shell pattern in the generated workflow that other tooling or reviewers would flag.

### Alternative 2: Sanitize/validate the repo name and keep inlining
Validate or escape the repository string and continue interpolating it directly into the shell command. Rejected because inlining runtime-resolved expressions into shell strings is the exact pattern that enables template injection; sanitizing an as-yet-unexpanded `${{ }}` expression at compile time is not feasible, and the safe, idiomatic GitHub Actions approach is env-var indirection.

## Consequences

### Positive
- The template-injection scanner passes on the compiler's own generated multi-repo steps; `target-repo` workflows compile again.
- Establishes a consistent, reusable pattern (env-var indirection) for safely passing expression-derived values into generated shell commands.

### Negative
- The generated YAML is more verbose: one extra `env:` entry per sub-repo plus per-repo references.
- Slightly more compiler logic and branching (literal vs. expression handling), increasing maintenance surface.

### Neutral
- Literal repository names are now quoted in the `env:` block rather than inlined; downstream behavior is unchanged.
- Four unit tests were added in `pkg/workflow/checkout_manager_test.go` to lock in single-repo, literal, expression-based (regression), and multi-sub-repo behavior.
